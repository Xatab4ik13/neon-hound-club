# План: починить оплату на всех устройствах

## Что выглядит сломанным сейчас

По коду уже видно 2 критические точки, которые вместе дают симптом «лоад-скрин → обратно в корзину/логин»:

1. **Серверный `/api/v1/payments/redirect` при любой проблеме не показывает явную ошибку, а уводит назад**
   - `server/src/routes/payments.ts:201-205, 244-305`
   - для заказа он делает `303` обратно на `/club/checkout?payment_error=...`
   - для неавторизованного юзера уводит на `/login`

2. **Чекаут-страница сама мгновенно выкидывает юзера в корзину, если `items.length === 0`**
   - `src/routes/club.checkout.tsx:116-119`
   - из-за этого даже если сервер вернул пользователя на checkout с ошибкой, UI может сразу перекинуть его в `/club/cart`, и реальная причина теряется

Дополнительно есть риск, что **кросс-доменные auth-cookie не доходят до `api.hhr.pro`** в реальном окружении/на части устройств:
- `server/src/lib/auth.ts:38-52`
- если кука не пришла, `/payments/redirect` отправит на `/login`, а пользователь увидит не «платёж не создался», а просто возврат назад

## Что сделаю

### 1. Разделю причину падения на 3 сценария
- **нет авторизации**
- **ошибка создания заказа**
- **ошибка инициализации платежа у банка**

И для каждого сценария сделаю явный возврат, чтобы пользователь видел конкретную причину, а не «магический откат».

### 2. Починю поведение checkout после возврата с ошибкой
В `src/routes/club.checkout.tsx`:
- уберу слишком ранний редирект в корзину при первом пустом состоянии
- не дам странице сразу выкидывать юзера, если он вернулся с `payment_error`
- сохраню и покажу ошибку стабильно, чтобы причина не исчезала из-за мгновенной навигации

### 3. Проверю и при необходимости поправлю auth/payment handoff
В payment flow:
- сверю, как фронт отправляет POST на `https://api.hhr.pro/api/v1/payments/redirect`
- проверю, что форма/JSON режим одинаково передают авторизацию
- при необходимости сделаю более надёжный сценарий для случая, когда сессия не дошла до API, чтобы пользователь видел нормальный state, а не бесконечный bounce

### 4. Зафиксирую один каноничный flow для всех устройств
Проверю и доведу до одного поведения:
- desktop browser
- mobile Safari/Chrome
- standalone PWA
- Hell Pass и checkout мерча

Цель: везде должен быть один понятный результат — либо открывается форма банка, либо пользователь остаётся на месте и видит точную ошибку.

## Технически затрону
- `src/routes/club.checkout.tsx`
- `src/routes/club.hell-pass.$tier.tsx` (если там тот же bounce-pattern)
- `server/src/routes/payments.ts`
- при необходимости один из вспомогательных payment/auth файлов, но без лишнего рефакторинга

## Как проверю после фикса

### Checkout товара
- авторизованный юзер с товаром в корзине
- клик «Оплатить»
- ожидаемый результат: либо банк открылся, либо checkout остался на месте и показал точную ошибку
- не должно быть немого возврата в корзину

### Hell Pass
- клик «Оплатить» по тиру
- ожидаемый результат: либо банк открылся, либо понятная ошибка без bounce-навигации

### Устройства
- desktop
- мобильный viewport
- PWA-flow не ломается
