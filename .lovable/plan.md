## Цель
Сделать один жёсткий, предсказуемый flow оплаты для мерча и Hell Pass, без «магии», без расхождений между устройствами и без лишних UI-экспериментов.

## Что сейчас ломает оплату
1. **У checkout слишком хрупкий источник данных.**
   В текущем flow заказ для оплаты собирается из данных формы, а не из одного серверного источника правды. Из-за этого backend может вернуть `payment_error=Корзина пустая`, даже когда пользователь только что жал «Оплатить».

2. **Слишком много развилок одного и того же сценария.**
   Сейчас есть разные ветки для:
   - checkout
   - Hell Pass
   - повторной оплаты заказа
   - PWA / не-PWA
   Это повышает шанс, что одна ветка работает, а соседняя нет.

3. **Ошибки backend маскируются как «отскок назад».**
   Пользователь видит сплэш/лоадер, а потом возврат назад, хотя реальная причина — отказ backend, пустая корзина, сессия или ошибка инициализации банка.

4. **Есть риск проблем с сессией между `hhr.pro` и `api.hhr.pro`.**
   Если cookie не доезжает стабильно в момент старта оплаты, backend не может продолжить flow и возвращает на логин/назад.

5. **UI платежа ушёл в сторону от продукта.**
   Кнопка с логотипами — это не ваш паттерн. Для проекта нужен обычный брендовый CTA «Оплатить», а логотипы — только как вторичная информация, если вообще нужны.

## Решение: как делаем правильно
### 1) Зафиксировать один канонический flow оплаты
Для мерча и для повторной оплаты заказа:
- пользователь жмёт **одну кнопку «Оплатить»**
- фронт отправляет только минимум обязательных данных
- backend сам создаёт/берёт заказ и сам инициирует банк
- фронт не должен собирать критичную платёжную логику из скрытых полей

Для Hell Pass:
- тот же принцип
- отдельный backend-путь допустим, но по тем же правилам: один submit, один ответ, одна явная ошибка

### 2) Сделать backend единственным источником правды
Для **checkout мерча**:
- состав заказа должен браться **из серверной корзины пользователя**, а не из hidden JSON в форме
- форма должна передавать только адрес/контакты/согласие и тип действия
- backend сам валидирует, что корзина не пуста, товар активен, размеры выбраны, остатки есть

Это убирает главный класс ошибок: «на фронте корзина есть, а backend увидел пусто».

### 3) Оставить один transport-механизм для старта оплаты
Не смешивать несколько полупохожих сценариев.

Каноника:
- **обычный браузер:** нативный POST формы в backend endpoint
- **standalone PWA:** backend возвращает `paymentUrl`, фронт ведёт на промежуточную `/pay/go`, где пользователь открывает банк через явную кнопку/ссылку

Важно: это не два разных бизнес-flow, а один flow с двумя способами навигации.

### 4) Разделить типы ошибок и показывать их явно
Нужны отдельные понятные сценарии:
- **сессия потеряна** → «Войди заново»
- **корзина пуста / товар недоступен / размер не выбран** → конкретное сообщение на checkout
- **банк не создал платёж** → конкретное сообщение, без отскока в никуда
- **платёж отклонён банком** → отдельный экран/статус с кнопкой повторить

Принцип:
- backend всегда возвращает конкретную причину
- frontend всегда показывает её прямо на странице, а не только toast

### 5) Упростить UI до вашего паттерна
- основная кнопка: **«Оплатить»** / **«Оплатить 12 900 ₽»**
- без логотипов внутри CTA
- если нужны знаки платёжных систем — они живут отдельно, как справочная строка, не как сама кнопка

### 6) Добить идемпотентность и повторную оплату
Для одного заказа нужна безопасная логика повторного запуска:
- если активный валидный платёж уже есть и ссылка жива — использовать его
- если предыдущая попытка упала — не зависать в страном промежуточном состоянии
- если провайдер вернул ошибку — платёж помечается как неуспешный явно, чтобы retry не блокировался

### 7) Проверить сессию и cookie как отдельный блок
Перед финальным запуском надо отдельно проверить:
- cookie домен
- `SameSite`
- `Secure`
- поведение при переходе `hhr.pro → api.hhr.pro → банк → hhr.pro`
- поведение в mobile Safari / Chrome / PWA

Пока это не подтверждено, нельзя считать оплату «готовой на 100%».

### 8) Финальная проверка не «на глаз», а по матрице сценариев
Проверяем минимум эти сценарии:

```text
1. Мерч / desktop browser / новый заказ
2. Мерч / mobile browser / новый заказ
3. Мерч / PWA / новый заказ
4. Мерч / повторная оплата существующего заказа
5. Hell Pass / desktop
6. Hell Pass / mobile
7. Hell Pass / PWA
8. Ошибка банка
9. Потеря сессии
10. Пустая корзина / невалидный товар / нет размера
```

Для каждого сценария должен быть один из двух результатов:
- либо открылся банк
- либо пользователь увидел **ясную человеческую ошибку на экране**

Не допускается третий вариант: «сплэш → непонятный возврат назад».

## Порядок реализации
1. **Заморозить scope** — не трогать дизайн, не трогать лишние фичи, не менять провайдера, не изобретать новый UX.
2. **Сделать один канонический backend flow для checkout мерча** от серверной корзины.
3. **Привести Hell Pass и повторную оплату заказа к тем же правилам обработки ошибок.**
4. **Вернуть обычную кнопку «Оплатить»** в ваш стиль.
5. **Добавить явные экранные ошибки** вместо немых возвратов.
6. **Проверить cookie/session handoff** между фронтом, API и банком.
7. **Прогнать матрицу сценариев** и только после этого считать задачу закрытой.

## Технические детали
- Backend Fastify остаётся центром всей платёжной логики.
- Для checkout нельзя опираться на hidden `items` как на критичный источник данных.
- Повторная оплата должна быть безопасной и не ломаться из-за старых `pending/new` записей.
- Экран `/pay/success` и `/pay/fail` должны быть следствием статуса платежа, а не способом угадывать, что произошло.
- UI редиректы не должны срабатывать раньше, чем догрузилось состояние пользователя и корзины.

## Что НЕ делаем
- не переделываем всё заново с нуля
- не меняем банк/эквайринг
- не добавляем новые платформенные сущности
- не делаем «умную» кнопку с логотипами, режимами, переключателями и прочими лишними идеями
- не считаем задачу решённой после одного удачного клика на одном устройстве
